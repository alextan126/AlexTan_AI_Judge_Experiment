题目：
公司 Web 端业务管理系统面临企业客户日志数据量激增的问题，现有单表查询响应缓慢。若需在 MySQL 中设计一种既能支持快速定位特定时间范围数据，又能避免全表扫描的存储方案，你会选择：A. 建立基于时间戳的复合唯一索引并配合 Redis 缓存热点数据；B. 采用分块查找思想将数据按时间分块存储，构建包含块最大时间戳和起始地址的二级索引表。请选择一种方案并简述理由。

模拟候选人回答：

我选方案 B。题目的核心问题是日志数据量激增导致单表查询响应缓慢，需要从存储层面根本性地解决。方案 B 将数据按时间分块存储并构建二级索引表，查询特定时间范围时先在轻量的索引表中定位到对应的数据块，再在块内检索，天然避免了全表扫描，查找效率为 O(√N) 级别，而且随着数据持续增长只需追加新的时间块和索引条目，扩展性好。方案 A 虽然复合索引也能加速查询，但日志数据量激增意味着单表持续膨胀，索引树本身会越来越庞大，维护成本上升；同时日志的时间戳并不唯一，用"复合唯一索引"在语义上不合理；Redis 缓存只能覆盖热点数据，对历史时间范围查询帮助有限。


备注：标准回答。选对方案 B，理由紧扣日志数据量激增、单表瓶颈、时间范围定位、避免全表扫描等原题场景，推导（分块查找复杂度分析、索引表轻量化、方案 A 的局限性）正确。

模拟候选人回答：

我选方案 B。日志数据量激增，单表扫描慢，需要分块来优化。方案 B 按时间分块存储后，每个块的数据量固定，查询时直接通过二级索引表的哈希定位到目标块，时间复杂度是 O(1)，比方案 A 的 B+ 树索引 O(log N) 还要快。而且分块存储后 MySQL 会自动将不活跃的块从内存中淘汰，所以基本不占内存，这是方案 A 加 Redis 做不到的。


备注：选项正确且挂钩了日志激增、单表慢查询等场景，但推导有两个事实错误：1）二级索引表的定位不是哈希 O(1)，而是顺序/二分查找 O(√N) 或 O(log M)；2）MySQL 不会"自动将不活跃块从内存淘汰"，块存储不等于自动内存管理优化。
