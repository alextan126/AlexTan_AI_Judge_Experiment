题目：
公司 Web 端业务管理系统面临企业客户日志数据量激增的问题，现有单表查询响应缓慢。若需在 MySQL 中设计一种既能支持快速定位特定时间范围数据，又能避免全表扫描的存储方案，你会选择：A. 建立基于时间戳的复合唯一索引并配合 Redis 缓存热点数据；B. 采用分块查找思想将数据按时间分块存储，构建包含块最大时间戳和起始地址的二级索引表。请选择一种方案并简述理由。


样本测试序号 A-01
STARE_level: STARE / score: 3

模拟候选人回答：
针对日志数据量激增导致单表查询缓慢的问题，需要设计一种能快速定位时间范围且避免全表扫描的方案。我选择方案 B。方案 B 采用分块查找思想，将数据按时间分块并构建二级索引表。查询时，先在轻量级的二级索引表中快速定位到目标时间范围所在的数据块，然后再进入具体块内检索，这天然避免了全表扫描，极大提升了检索效率。而方案 A 存在明显缺陷：首先，日志的时间戳通常不是唯一的，建立“唯一索引”在业务语义上是错误的；其次，随着海量日志的持续写入，单表 B+ 树索引会变得极其庞大，维护成本极高；最后，Redis 缓存虽然能加速热点数据，但对于历史时间范围的日志排查毫无帮助。因此，方案 B 是从存储结构底层解决海量日志时间范围查询的最佳选择。

完美的标准答案。包含了 S/T（转述问题：日志激增，单表查询慢，需按时间范围定位），A（选择方案 B），R（方案 B 通过二级索引定位数据块，避免全表扫描），E（深入分析了方案 A 的致命缺陷：时间戳不唯一、单表索引膨胀、Redis 无法解决历史查询，从而反证方案 B 的合理性）。

模拟候选人回答：
对于日志系统，最好的方案是直接接入 ELK (Elasticsearch, Logstash, Kibana) 技术栈。MySQL 根本就不适合存日志，不管怎么分块或者加索引，最后都会遇到性能瓶颈。用 Elasticsearch 做倒排索引，查询速度快，而且自带 Kibana 可以直接做可视化报表，这才是业界主流的做法。
